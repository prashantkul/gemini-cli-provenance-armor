# Gemini CLI: Audit Logging & Traceability Guide

This document details how to interpret Gemini CLI's internal logs to distinguish between user-provided instructions and the autonomous actions (tool calls) performed by the CLI.

## 1. Log Locations

Gemini CLI maintains logs in two primary locations:
*   **Global Telemetry:** `~/.gemini/logs.json` (Cumulative history across all projects).
*   **Session-Specific:** `~/.gemini/tmp/<project_id>/logs.json` (Detailed logs for the current active session).
*   **Chat History:** `~/.gemini/tmp/<project_id>/chats/history.json` (Structured JSON array of the full conversation).

## 2. Event Schema (OpenTelemetry Format)

Logs follow the **OpenTelemetry LogRecord** JSON schema. Each entry contains a `name` field that identifies the event type.

### A. User Actions (`gemini_cli.user_prompt`)
These events represent the direct input provided by the user.

| Attribute | Description |
| :--- | :--- |
| `name` | `gemini_cli.user_prompt` |
| `attributes.prompt` | The literal text typed by the user (if `logPrompts: true` in settings). |
| `attributes.prompt_id` | A unique UUID used to link this instruction to subsequent model actions. |
| `attributes.prompt_length` | Character count of the user input. |

### B. Gemini CLI Actions (`gemini_cli.tool_call`)
These events record every time the model decides to use a tool (e.g., read a file, run a command).

| Attribute | Description |
| :--- | :--- |
| `name` | `gemini_cli.tool_call` |
| `attributes.function_name` | The specific tool invoked (e.g., `run_shell_command`, `read_file`). |
| `attributes.function_args` | The exact JSON arguments generated by the model for the tool. |
| `attributes.decision` | `accept` (User approved), `reject` (User denied), or `auto_accept` (YOLO mode). |
| `attributes.success` | Boolean indicating if the tool execution was successful. |

### C. System Operations (`gemini_cli.file_operation`)
A specialized log for granular file system changes.

| Attribute | Description |
| :--- | :--- |
| `name` | `gemini_cli.file_operation` |
| `attributes.operation` | `create`, `read`, `update`, or `delete`. |
| `attributes.path` | The absolute path of the file being modified. |

## 3. Distinguishing "Ask" vs. "Action"

| Feature | User "Ask" (Instruction) | Gemini CLI "Action" (Execution) |
| :--- | :--- | :--- |
| **Origin** | Human | Generative Model |
| **Format** | Natural Language | Structured JSON |
| **Event Name** | `gemini_cli.user_prompt` | `gemini_cli.tool_call` |
| **Validation** | N/A | Subject to `.toml` Policy Engine |
| **Visibility** | Always visible in terminal | Often backgrounded unless `--verbose` |

## 4. Analysis via CLI Tools

You can use `jq` to extract a clean audit trail from the log files:

```bash
# List all autonomous actions and their arguments
jq '.[] | select(.name == "gemini_cli.tool_call") | {tool: .attributes.function_name, args: .attributes.function_args, status: .attributes.decision}' ~/.gemini/logs.json

# Trace a specific user prompt to its resulting tool calls
jq '.[] | select(.attributes.prompt_id == "TARGET_PROMPT_ID")' ~/.gemini/logs.json
```

## 5. Security Implications
*   **Traceability:** Every file change can be traced back to the specific `tool_call` that performed it, and that `tool_call` is linked via `prompt_id` to the original user instruction.
*   **Auditability:** In enterprise environments, these logs can be exported via OpenTelemetry to centralized logging systems (e.g., Google Cloud Logging) for security monitoring and compliance.
